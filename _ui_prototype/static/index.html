What is this?

fd-eval-harness is a shared execution layer for evaluating full-duplex speech-to-speech (STS) AI models. You give it an audio file and ground-truth labels, and it runs a model adapter against a scoring task to produce a single, comparable score.

Different STS models have different I/O shapes. The harness hides that behind a common runner, so the same task can score Moshi, a stub, or a real-time cloud model the same way.

Core concepts
▶ Task what is measured
Picks a scoring rule. Example tasks: voice_activity_detection, turn_taking_latency, tool_use_under_disfluency, interruption_state_update.
⚙ Adapter the model under test
Wraps a concrete model (or a test stub) so it emits prediction events in a uniform schema. Examples: energy_vad, moshi, tool_use_stub.
◎ AudioSession the conversation
A multichannel audio blob plus which channels are input (what the model hears) vs. target (what the model is expected to produce on).
⟋ Predictions ⟍ References
The adapter streams PredictionEvents (VAD state changes, turn onsets, tool calls, transcript chunks). Your labels.json becomes typed References. The task compares the two and returns a score.
How to use (3 steps)
  1. 1
    Pick a Task and an Adapter on the left. The chips tell you what kind of measurement this is and what kind of model is being tested.
  2. 2
    Upload audio.wav and labels.json, or hit Load sample pair for a canned example. Set which channels the model listens on (In) and which it answers on (Tgt).
  3. 3
    Hit RUN EVALUATION. You get back a score in [0, 1], per-task details, and a preview of the first 20 predictions/references.
What happens when you press RUN
1. UI uploads audio + labels to /api/evaluate.
2. Server builds an AudioSession with your channel config.
3. The adapter streams prediction events while processing the session.
4. The task parses your labels into typed References, then compares predictions against them.
5. You get back a TaskResult: score (0–1) + details (precision/recall, token counts, judge reasoning, …).
Chip legend
◎ observer task grades the audio, model is passive ▶ participant model must actually respond algorithmic deterministic scoring (F1, precision, …) llm-judge LLM grades the response (needs OPENAI_API_KEY)
